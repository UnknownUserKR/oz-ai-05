# 오늘 학습한 내용 정리
오늘은 Git & Github와 HTML 기초에 대한 내용을 공부하였습니다!

## Github

- Git & Github에 대해서 2일차로 실제 Github에 연결해서 Push & Pull 하는 방법에 대해서 자세히 배워봤습니다.
    - 협업 준비: 계정, 인증, 권한, 저장소 초대
    - 작업 시작: Issue 선택, 브랜치 생성, commit/push
    - PR 생성: PR 템플릿, 관련 Issue 연결, 리뷰 요청
    - 리뷰와 병합: approve/request changes, conflict 해결, merge 방식

### 1. Github Flow

Github Flow는 `main` 브랜치를 안정적인 상태로 유지하면서, 새로운 기능이나 수정 작업은 별도의 브랜치에서 진행하고 Pull Request를 통해 병합하는 협업 방식입니다.

#### 기본 흐름

1. Issue 또는 작업 내용을 확인한다.
2. 작업용 브랜치를 생성한다.
3. 코드를 수정하고 커밋한다.
4. 원격 저장소에 브랜치를 push한다.
5. Github에서 Pull Request를 생성한다.
6. 코드 리뷰를 받고 수정 사항을 반영한다.
7. 리뷰와 테스트가 완료되면 `main` 브랜치에 merge한다.
8. 병합 후 작업 브랜치를 삭제한다.

#### 예시 명령어

```bash
git switch -c feature/login-page
git add .
git commit -m "Add login page"
git push origin feature/login-page
```

#### 팀 규칙 예시

- `main` 브랜치에는 직접 push하지 않는다.
- 모든 작업은 별도 브랜치에서 진행한다.
- Pull Request는 최소 1명 이상의 리뷰를 받은 뒤 merge한다.
- merge 후 사용하지 않는 브랜치는 삭제한다.

### 2. 인증과 접속 방식

Github에서 저장소를 사용하려면 웹 로그인뿐 아니라 Git 명령어를 사용할 때의 인증 방식도 알아야 합니다. 현재 Github는 보안상 계정 비밀번호만으로 Git 명령어 인증을 하지 않으며, HTTPS 방식에서는 Personal Access Token을, SSH 방식에서는 SSH Key를 사용합니다.

| 방식 | 설명 | 사용 상황 |
|---|---|---|
| Web 로그인 | Github 사이트에 접속할 때 사용하는 로그인 | Issue, PR, Review 작성 |
| HTTPS + Token | 저장소 주소가 `https://`로 시작하며 토큰으로 인증 | 초보자 또는 간단한 설정 |
| SSH Key | 내 컴퓨터의 SSH Key를 Github에 등록해서 인증 | 자주 push/pull 하는 개발 환경 |

#### HTTPS 예시

```bash
git clone https://github.com/username/repository.git
```

#### 주의할 점

- 비밀번호, 토큰, SSH private key는 절대 저장소에 올리지 않는다.
- `.env` 파일에는 API Key나 비밀번호가 들어갈 수 있으므로 `.gitignore`에 추가한다.
- 가능하면 Github 계정에 2단계 인증을 설정한다.

### 3. 브랜치 이름 규칙

브랜치 이름은 어떤 작업을 하는지 한눈에 알 수 있도록 작성해야 합니다.

#### 권장 형식

```text
작업유형/작업-설명
```

#### 브랜치 이름 예시

| 유형 | 의미 | 예시 |
|---|---|---|
| `feature/` | 새로운 기능 추가 | `feature/login-page` |
| `fix/` | 버그 수정 | `fix/login-error` |
| `docs/` | 문서 수정 | `docs/update-readme` |
| `refactor/` | 코드 구조 개선 | `refactor/user-service` |
| `hotfix/` | 긴급 수정 | `hotfix/payment-bug` |
| `test/` | 테스트 코드 추가/수정 | `test/login-api` |

#### 팀 규칙 예시

- 브랜치 이름은 영어 소문자와 하이픈을 사용한다.
- 공백, 한글, 특수문자는 사용하지 않는다.
- 하나의 브랜치에는 하나의 목적만 담는다.

### 4. Issue 작성 방법

Issue는 해야 할 일, 버그, 기능 요청, 논의할 내용을 기록하는 공간입니다. 팀원이 Issue만 보고도 어떤 작업인지 이해할 수 있게 작성해야 합니다.

#### Issue 종류

| 종류 | 목적 |
|---|---|
| Bug | 오류나 문제 보고 |
| Feature | 새로운 기능 제안 |
| Task | 해야 할 작업 정리 |
| Docs | 문서 수정 |
| Question | 질문 또는 논의 |

#### Issue 작성 예시

```md
## 작업 내용
로그인 페이지에서 이메일과 비밀번호를 입력받는 UI를 구현합니다.

## 필요한 이유
사용자가 서비스에 로그인할 수 있어야 하기 때문입니다.

## 완료 조건
- [ ] 이메일 입력창 생성
- [ ] 비밀번호 입력창 생성
- [ ] 로그인 버튼 생성
- [ ] 빈 값 입력 시 에러 메시지 출력

## 참고 자료
관련 화면 또는 디자인 링크를 첨부합니다.
```

#### 팀 규칙 예시

- 작업을 시작하기 전에 Issue를 생성하거나 기존 Issue를 선택한다.
- 담당자, label, milestone을 가능하면 지정한다.
- PR을 만들 때 관련 Issue 번호를 연결한다.

### 5. Pull Request 작성 방법

Pull Request는 내가 작업한 내용을 `main` 브랜치에 반영해 달라고 요청하는 과정입니다. 리뷰어가 변경 내용을 빠르게 이해할 수 있도록 작성해야 합니다.

#### PR 작성 예시

```md
## 변경 내용
- 로그인 페이지 UI 추가
- 이메일/비밀번호 입력창 추가
- 빈 값 입력 시 에러 메시지 출력

## 관련 Issue
Closes #12

## 테스트 방법
1. 로그인 페이지에 접속합니다.
2. 이메일과 비밀번호를 입력합니다.
3. 로그인 버튼을 클릭합니다.
4. 에러 메시지 또는 성공 동작을 확인합니다.

## 리뷰 요청 사항
- 에러 메시지 처리 방식이 적절한지 확인 부탁드립니다.

## 스크린샷
필요한 경우 화면 이미지를 첨부합니다.
```

#### 팀 규칙 예시

- PR 제목은 작업 내용을 명확하게 작성한다.
- 하나의 PR에는 하나의 목적만 담는다.
- 변경 내용, 테스트 방법, 관련 Issue를 작성한다.
- `Closes #이슈번호`를 사용하면 PR merge 시 Issue가 자동으로 닫힌다.

### 6. 코드 리뷰 방법

코드 리뷰는 병합 전에 코드 품질을 확인하고 문제를 미리 발견하기 위한 과정입니다.

#### 리뷰 상태

| 상태 | 의미 |
|---|---|
| Comment | 의견만 남김 |
| Approve | 병합해도 좋음 |
| Request changes | 수정이 필요함 |

#### 리뷰할 때 확인할 내용

- 요구사항을 제대로 구현했는가?
- 버그가 발생할 가능성은 없는가?
- 코드가 너무 복잡하지 않은가?
- 변수명, 함수명이 이해하기 쉬운가?
- 중복 코드가 많지 않은가?
- 테스트가 필요한 부분은 없는가?
- 비밀번호, 토큰, API Key 같은 민감 정보가 포함되지 않았는가?

#### 좋은 리뷰 예시

```text
이 조건문은 사용자가 로그인하지 않은 상태에서도 실행될 수 있어 보입니다.
로그인 여부를 먼저 확인하는 로직을 추가하면 더 안전할 것 같습니다.
```

#### 팀 규칙 예시

- 코드가 아니라 변경 내용에 대해 리뷰한다.
- 수정 요청을 할 때는 이유를 함께 적는다.
- PR 작성자는 리뷰를 반영한 뒤 resolved 처리한다.
- 리뷰가 끝난 뒤 approve를 받으면 merge한다.

### 7. CODEOWNERS

CODEOWNERS는 특정 파일이나 폴더의 담당자를 지정하는 Github 기능입니다. 지정된 영역이 수정되면 담당자가 자동으로 리뷰어로 요청됩니다.

#### CODEOWNERS 파일 위치

```text
.github/CODEOWNERS
CODEOWNERS
docs/CODEOWNERS
```

#### 작성 예시

```txt
# 전체 파일 기본 담당자
* @team-lead

# 프론트엔드 담당자
/frontend/ @frontend-team

# 백엔드 담당자
/backend/ @backend-team

# 문서 담당자
/docs/ @docs-team

# 특정 파일 담당자
README.md @username
```

#### 사용하는 이유

- 중요한 코드가 담당자 확인 없이 merge되는 것을 방지할 수 있다.
- 폴더별 책임자를 명확히 할 수 있다.
- PR 생성 시 리뷰어를 자동으로 지정할 수 있다.

#### 팀 규칙 예시

- 중요한 폴더에는 CODEOWNERS를 지정한다.
- `main` 브랜치 보호 규칙에서 Code Owner 승인을 필수로 설정한다.
- 담당자가 바뀌면 CODEOWNERS 파일도 함께 수정한다.

---

### 정리

Github 협업에서는 단순히 push와 pull을 하는 것뿐만 아니라, Issue로 작업을 관리하고, 브랜치에서 작업한 뒤 Pull Request를 생성하고, 코드 리뷰를 통해 안정적으로 main 브랜치에 병합하는 흐름이 중요합니다.

특히 팀 프로젝트에서는 브랜치 이름 규칙, Issue 작성 방식, PR 작성 방식, 리뷰 기준, CODEOWNERS 같은 규칙을 미리 정해두면 협업 과정에서 혼란을 줄일 수 있습니다.


## HTML

- 오늘 HTML의 기본 세팅에 대해 간단하게 배워봤습니다.

### HTML의 기본 구성

- `<!DOCTYPE html>` : HTML 문서라는 것을 알리는 선언문
- `<html>` : 실제 HTML 문서를 감싸는 최상위 태그
- `<head>` : 웹사이트의 제목, 설정, 메타 정보 등을 작성하는 부분
- `<body>` : 실제로 화면에 보여지는 본문 내용을 작성하는 부분
- `<footer>` : 페이지 맨 아래에 저작권, 연락처, 기타 정보를 작성하는 부분

### HTML 내부 세부 작성할 때 사용하는 코드

- `<h1>` : 가장 큰 제목을 작성할 때 사용하는 태그
- `<p>` : 본문 문단을 작성할 때 사용하는 태그
- `<hr>` : 문단을 구분하는 가로선을 만들 때 사용하는 태그
- `<br>` : 줄바꿈을 하고 싶을 때 사용하는 태그
